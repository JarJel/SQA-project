# 3. Rencana Pengujian — Midnight Finance

Bab ini merinci peta jalan (*roadmap*) pelaksanaan pengujian **Sistem Manajemen Keuangan Pribadi Midnight Finance**. Rencana ini disusun untuk memastikan setiap komponen diuji secara sistematis menggunakan metode **White Box Testing**, **Black Box Testing**, dan **Gray Box Testing**, sesuai taksonomi pengujian perangkat lunak (Suprihadi, 2025).

---

## 3.1 Strategi Fase Pengujian

Pengujian Midnight Finance dibagi dalam tiga fase utama berdasarkan jenis teknik pengujian:

### Fase I: White Box Testing
Pengujian dilakukan dengan memeriksa logika internal *source code* backend Laravel menggunakan teknik **Basis Path Testing**.

- **Fokus:** Fungsi autentikasi (`AuthController`), pencatatan transaksi (`TransactionController`), manajemen kategori (`CategoryController`)
- **Target:** Memetakan semua jalur eksekusi, menghitung Cyclomatic Complexity (V(G)), dan memastikan setiap jalur tereksekusi minimal satu kali
- **Penanggung Jawab:** Backend Developer & QA Engineer

### Fase II: Black Box Testing
Pengujian dilakukan pada antarmuka sistem (API endpoint) tanpa melihat kode internal, menggunakan teknik **Equivalence Partitioning** dan **Boundary Value Analysis**.

- **Fokus:** Validasi input registrasi, login, input nominal transaksi, manajemen kategori
- **Target:** Memastikan sistem menolak input invalid dan menerima input valid sesuai spesifikasi
- **Penanggung Jawab:** QA Team

### Fase III: Gray Box Testing
Kombinasi antara White Box dan Black Box, menggunakan teknik **Regression Testing** dan **Pattern Testing** untuk memastikan tidak ada fitur yang rusak setelah perubahan kode.

- **Fokus:** Sinkronisasi saldo otomatis setelah transaksi, konsistensi data lintas modul
- **Target:** Memastikan integrasi backend-frontend berjalan benar dan tidak ada regresi
- **Penanggung Jawab:** QA Team & Backend Developer

## 3.2 Kriteria Masuk & Keluar (Entry & Exit Criteria)

| Kriteria | Deskripsi |
|:---|:---|
| **Entry Criteria** | 1. Backend Laravel sudah berjalan di lingkungan lokal/staging<br>2. Database sudah di-migrate dan di-seed dengan data uji<br>3. Dokumen Test Case sudah disetujui tim |
| **Exit Criteria** | 1. Seluruh *Critical* dan *High Priority* bug sudah diperbaiki<br>2. Semua Test Case telah dieksekusi dengan hasil tercatat<br>3. Laporan pengujian final telah disetujui ketua tim |

## 3.3 Jadwal Eksekusi Pengujian

| ID | Aktivitas Pengujian | Modul yang Diuji | Teknik | Prioritas |
|:---:|:---|:---|:---:|:---:|
| T-01 | White Box — Login | `AuthController@login` | Basis Path Testing | 🔴 High |
| T-02 | White Box — Catat Transaksi | `TransactionController@store` | Basis Path Testing | 🔴 High |
| T-03 | White Box — Kategori | `CategoryController@store` | Basis Path Testing | 🟡 Medium |
| T-04 | Black Box — Registrasi | `POST /api/register` | Equivalence Partitioning | 🔴 High |
| T-05 | Black Box — Nominal Transaksi | `POST /api/transactions` (`amount`) | Boundary Value Analysis | 🔴 High |
| T-06 | Black Box — Login | `POST /api/login` | Decision Table Testing | 🟡 Medium |
| T-07 | Gray Box — Saldo Otomatis | `TransactionController` + `FinancialAccount` | Regression Testing | 🔴 High |
| T-08 | Gray Box — Riwayat Transaksi | `GET /api/transactions` (filter) | Pattern Testing | 🟡 Medium |
| T-09 | Bug Fixing | Seluruh modul | — | 🔴 High |
| T-10 | Regression setelah Fix | Seluruh modul | Regression Testing | 🟡 Medium |
