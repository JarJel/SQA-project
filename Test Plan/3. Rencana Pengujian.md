# 3. Rencana Pengujian (Comprehensive Test Schedule & Strategy)

Bab ini merinci peta jalan (*roadmap*) pelaksanaan pengujian Sistem Pemesanan Café Berbasis Web. Rencana ini disusun untuk memastikan setiap komponen diuji secara sistematis, mulai dari unit terkecil hingga integrasi sistem secara menyeluruh, guna menjamin stabilitas aplikasi sebelum tahap *Go-Live*.

---

## 3.1 Strategi Fase Pengujian

Pengujian tidak dilakukan secara acak, melainkan dibagi menjadi lima fase utama yang saling berkesinambungan. Setiap fase memiliki fokus yang berbeda untuk meminimalisir risiko kegagalan sistem.

### Fase I: Unit Testing (Pengujian Unit)
Fase ini berfokus pada pengujian komponen atau fungsi terkecil dari kode program secara terisolasi.
* **Fokus:** Validasi logika fungsi, algoritma perhitungan harga, dan skrip validasi form.
* **Target:** Memastikan fungsi seperti `calculateTotal()`, `validateEmail()`, dan `checkStock()` berjalan benar sebelum digabungkan.
* **Penanggung Jawab:** Backend & Frontend Developer.

### Fase II: Integration Testing (Pengujian Integrasi)
Setelah unit terkecil dinyatakan lulus, pengujian dilakukan pada gabungan antar modul untuk melihat bagaimana mereka berinteraksi.
* **Fokus:** Aliran data antara modul *Cart* ke *Checkout*, serta koneksi antara aplikasi web dengan database API.
* **Target:** Memastikan data yang dikirim dari antarmuka pengguna tersimpan dengan benar di database server tanpa adanya distorsi data.
* **Penanggung Jawab:** QA Engineer & Backend Developer.

### Fase III: System Testing (Pengujian Sistem)
Pengujian dilakukan pada sistem secara utuh (*End-to-End*) untuk memverifikasi apakah sistem memenuhi kebutuhan fungsional yang telah ditetapkan.
* **Fokus:** Seluruh alur kerja dari login pengguna hingga status pesanan diubah oleh admin.
* **Target:** Menemukan *bug* yang hanya muncul saat semua komponen sistem berjalan bersamaan dalam lingkungan staging.
* **Penanggung Jawab:** QA Team.

### Fase IV: User Acceptance Testing (UAT)
Pengujian akhir yang dilakukan oleh pengguna akhir (pemilik café atau staf kasir) untuk memvalidasi apakah aplikasi sudah sesuai dengan proses bisnis nyata.
* **Fokus:** Kemudahan penggunaan (*Usability*) dan kesesuaian fitur dengan kebutuhan operasional café sehari-hari.
* **Target:** Mendapatkan persetujuan (*sign-off*) dari pemilik proyek untuk rilis.
* **Penanggung Jawab:** Product Owner & Stakeholder.

---

## 3.2 Kriteria Masuk & Keluar (Entry & Exit Criteria)

Untuk menjaga kualitas, setiap fase pengujian harus memenuhi syarat berikut:

| Kriteria | Deskripsi |
| :--- | :--- |
| **Entry Criteria** | 1. Kode program sudah di-*deploy* ke server staging.<br>2. Data pengujian (menu, user) sudah siap.<br>3. Dokumen *Test Case* sudah disetujui. |
| **Exit Criteria** | 1. Semua *Critical* dan *High Priority* bug sudah diperbaiki dan diuji ulang.<br>2. Seluruh *Test Case* telah dieksekusi.<br>3. Laporan pengujian akhir telah disetujui oleh tim pimpinan. |

---

## 3.3 Jadwal Eksekusi Pengujian (Detailed Schedule)

Berikut adalah tabel matriks waktu pelaksanaan pengujian secara mendetail:

| ID Tahap | Aktivitas Pengujian | Komponen Utama yang Diuji | Estimasi Waktu | Prioritas |
| :--- | :--- | :--- | :--- | :--- |
| **T-01** | **Unit Testing** | Logic validasi form, algoritma diskon, dan fungsi database. | 3 Hari | High |
| **T-02** | **Integration Testing** | Sinkronisasi database, koneksi Payment Gateway, API response. | 4 Hari | High |
| **T-03** | **Security Smoke Test** | Proteksi URL admin, validasi session login, pencegahan XSS. | 2 Hari | Medium |
| **T-04** | **System Testing** | Alur lengkap pemesanan (E2E), responsivitas Mobile/Desktop. | 5 Hari | High |
| **T-05** | **UAT (User Acceptance)** | Simulasi kasir nyata, pengujian alur pesanan oleh pemilik café. | 2 Hari | Critical |
| **T-06** | **Bug Fixing** | Perbaikan temuan bug dari fase T-01 sampai T-05. | 3 Hari | High |
| **T-07** | **Regression Testing** | Pengujian ulang setelah bug fix untuk menjamin tidak ada error baru. | 2 Hari | Medium |
| **T-08** | **Reporting** | Penyusunan laporan hasil akhir dan dokumentasi bug. | 1 Hari | Low |

---

## 3.4 Deliverables (Hasil Akhir)

Setelah seluruh rencana pengujian ini dilaksanakan, tim penguji wajib menghasilkan dokumen-dokumen berikut:
1.  **Test Case Document:** Kumpulan detail skenario yang telah dijalankan.
2.  **Bug Report:** Daftar temuan kesalahan sistem beserta tingkat keparahannya.
3.  **Test Summary Report:** Ringkasan hasil pengujian yang menyatakan apakah aplikasi layak rilis atau tidak.
