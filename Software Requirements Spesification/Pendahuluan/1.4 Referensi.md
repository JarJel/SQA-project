## 1.4 Referensi *(References)*

### 1.4.1 Tujuan Bagian Ini

Bagian ini mendokumentasikan seluruh sumber eksternal yang dijadikan acuan dalam penyusunan dokumen SRS ini maupun dalam implementasi teknis aplikasi **Sapopoe**. Referensi yang tercantum mencakup standar internasional, dokumentasi resmi teknologi yang digunakan, serta pustaka pendukung yang mempengaruhi keputusan desain dan arsitektur sistem.

Seluruh referensi disusun berdasarkan kategori untuk memudahkan penelusuran. Tautan (*URL*) yang dicantumkan merujuk pada versi dokumentasi resmi yang tersedia secara publik dan dianggap valid pada saat dokumen ini disusun. Tim pengembang bertanggung jawab untuk memverifikasi kesesuaian versi dokumentasi dengan versi *dependency* yang aktif digunakan dalam *codebase* proyek.

---

### 1.4.2 Standar Internasional *(International Standards)*

Referensi berikut merupakan standar formal yang dijadikan landasan metodologis dalam penyusunan dokumen SRS ini serta dalam penilaian kualitas sistem Sapopoe.

---

#### [REF-STD-01] IEEE 830-1998

| Atribut | Detail |
|---|---|
| **Judul Lengkap** | *IEEE Recommended Practice for Software Requirements Specifications* |
| **Penerbit** | Institute of Electrical and Electronics Engineers (IEEE) |
| **Tahun Terbit** | 1998 |
| **Nomor Standar** | IEEE Std 830-1998 |
| **Status** | Superseded — tetap dirujuk sebagai praktik terbaik penulisan SRS |
| **Relevansi** | Dokumen SRS ini mengikuti struktur, konvensi penamaan bagian, dan pedoman penulisan kebutuhan fungsional maupun non-fungsional sebagaimana direkomendasikan oleh standar ini. Standar IEEE 830-1998 mendefinisikan karakteristik SRS yang baik: *correct, unambiguous, complete, consistent, ranked, verifiable, modifiable,* dan *traceable.* |
| **Akses Resmi** | [https://ieeexplore.ieee.org/document/720574](https://ieeexplore.ieee.org/document/720574) |

---

#### [REF-STD-02] ISO/IEC 25010:2011

| Atribut | Detail |
|---|---|
| **Judul Lengkap** | *Systems and Software Engineering — Systems and Software Quality Requirements and Evaluation (SQuaRE) — System and Software Quality Models* |
| **Penerbit** | International Organization for Standardization (ISO) / International Electrotechnical Commission (IEC) |
| **Tahun Terbit** | 2011 |
| **Nomor Standar** | ISO/IEC 25010:2011 |
| **Status** | Aktif |
| **Relevansi** | Standar ini mendefinisikan model kualitas perangkat lunak yang digunakan sebagai kerangka evaluasi non-fungsional sistem Sapopoe. Tiga karakteristik kualitas yang secara khusus dirujuk dalam dokumen ini adalah: **(1) Security** — mencakup kerahasiaan data, integritas, dan autentikasi pengguna; **(2) Usability** — mencakup kemudahan penggunaan bagi seluruh segmen pengguna; dan **(3) Performance Efficiency** — mencakup waktu respons sistem dan efisiensi penggunaan sumber daya dalam kondisi multi-sesi. |
| **Akses Resmi** | [https://www.iso.org/standard/35733.html](https://www.iso.org/standard/35733.html) |

### 1.4.3 Dokumentasi Teknologi Backend *(Backend Technology References)*

Referensi berikut berkaitan dengan teknologi yang digunakan pada sisi server (*server-side*) aplikasi Sapopoe.

---

#### [REF-BE-01] Laravel Framework

| Atribut | Detail |
|---|---|
| **Nama Teknologi** | Laravel |
| **Deskripsi** | *The PHP Framework for Web Artisans* — framework backend berbasis PHP yang digunakan sebagai fondasi utama sisi server aplikasi Sapopoe, mencakup routing, ORM (*Eloquent*), manajemen migrasi basis data, validasi, dan sistem antrian. |
| **Versi Rujukan** | Laravel 11.x |
| **Penerbit / Pengelola** | Taylor Otwell & Laravel LLC |
| **Lisensi** | MIT License |
| **Dokumentasi Resmi** | [https://laravel.com/docs](https://laravel.com/docs) |
| **Repositori** | [https://github.com/laravel/laravel](https://github.com/laravel/laravel) |
| **Relevansi** | Seluruh logika bisnis backend Sapopoe — termasuk pengelolaan transaksi, kalkulasi saldo, manajemen anggaran (*budgeting*), dan mekanisme penghapusan akun terjadwal — diimplementasikan menggunakan fitur-fitur inti Laravel. |

---

#### [REF-BE-02] Laravel Sanctum

| Atribut | Detail |
|---|---|
| **Nama Teknologi** | Laravel Sanctum |
| **Deskripsi** | Paket autentikasi resmi dari ekosistem Laravel yang menyediakan sistem manajemen *Personal Access Token* (PAT) berbasis API, dirancang khusus untuk aplikasi *single-page application* (SPA) dan API mobile. |
| **Versi Rujukan** | Laravel Sanctum 4.x (bagian dari Laravel 11.x) |
| **Penerbit / Pengelola** | Taylor Otwell & Laravel LLC |
| **Lisensi** | MIT License |
| **Dokumentasi Resmi** | [https://laravel.com/docs/sanctum](https://laravel.com/docs/sanctum) |
| **Repositori** | [https://github.com/laravel/sanctum](https://github.com/laravel/sanctum) |
| **Relevansi** | Laravel Sanctum merupakan fondasi teknis dari seluruh mekanisme autentikasi Sapopoe, mencakup: penerbitan token per perangkat, validasi token pada setiap *request* API, pencabutan token saat *session timeout* (24 jam inaktivitas), pencabutan seluruh token saat konfirmasi penghapusan akun, serta dukungan *concurrent multi-device access* melalui manajemen multi-token per pengguna. |

---

#### [REF-BE-03] Intervention Image

| Atribut | Detail |
|---|---|
| **Nama Teknologi** | Intervention Image |
| **Deskripsi** | Pustaka pemrosesan gambar berbasis PHP yang menyediakan antarmuka ekspresif untuk manipulasi gambar, termasuk pengubahan ukuran (*resizing*), pemotongan (*cropping*), dan konversi format gambar secara server-side. |
| **Versi Rujukan** | Intervention Image 3.x |
| **Penerbit / Pengelola** | Oliver Vogel (olivervogel) |
| **Lisensi** | MIT License |
| **Dokumentasi Resmi** | [https://image.intervention.io](https://image.intervention.io) |
| **Repositori** | [https://github.com/Intervention/image](https://github.com/Intervention/image) |
| **Relevansi** | Digunakan secara spesifik dalam pipeline pemrosesan foto profil pengguna Sapopoe. Setiap gambar yang diunggah oleh pengguna — terlepas dari format aslinya (JPEG, PNG, GIF, dll.) — diproses oleh Intervention Image untuk dikonversi ke format **WebP** sebelum disimpan di server, guna mengoptimalkan efisiensi penyimpanan dan kecepatan pengiriman aset. |

### 1.4.6 Catatan Pemeliharaan Referensi

> **Penting bagi Tim Pengembang (REMAKO):**
>
> Dokumentasi teknologi bersifat dinamis dan dapat berubah seiring rilis versi baru. Tim pengembang bertanggung jawab untuk:
>
> 1. **Memverifikasi kompatibilitas versi** setiap *dependency* yang tercantum dengan versi yang aktif digunakan dalam `composer.json` dan `package.json` proyek.
> 2. **Memperbarui kode referensi** pada tabel di atas apabila terjadi *major version upgrade* yang mempengaruhi perilaku sistem yang terdokumentasi dalam SRS ini.
> 3. **Mendokumentasikan *breaking changes*** dari versi referensi yang memerlukan penyesuaian implementasi sebagai lampiran revisi dokumen SRS.
