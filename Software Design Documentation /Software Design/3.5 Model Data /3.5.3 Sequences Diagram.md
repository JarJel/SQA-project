# 3.5.3 Sequence Diagram

*Sequence Diagram* ini menggambarkan interaksi antar objek-objek dalam sistem **Sapopoe** secara terperinci dan berurutan berdasarkan waktu. Diagram ini memiliki dua dimensi: **dimensi vertikal** yang menunjukkan urutan waktu, dan **dimensi horizontal** yang menunjukkan objek-objek yang terlibat.

Objek utama yang terlibat di setiap alur adalah:
- **User** — Aktor utama yang berinteraksi dengan sistem
- **Frontend (React)** — Antarmuka pengguna
- **API Controller (Laravel)** — Pengendali logika bisnis di sisi server
- **Middleware / Service** — Lapisan layanan pendukung (Auth, Mail, Image)
- **Database (MySQL)** — Penyimpanan data permanen

---

## Keterangan Simbol

| Simbol | Nama | Keterangan |
|---|---|---|
| Kotak persegi panjang di atas | **Objek / Lifeline Head** | Merepresentasikan objek atau aktor yang terlibat |
| Garis putus-putus vertikal | **Lifeline** | Menggambarkan keberadaan objek selama interaksi berlangsung |
| Kotak tipis pada lifeline | **Activation Box** | Menunjukkan periode waktu objek sedang aktif memproses |
| Panah solid `->>`  | **Synchronous Message** | Pesan sinkron — pengirim menunggu balasan sebelum lanjut |
| Panah putus `-->>`  | **Return Message** | Pesan balasan dari objek yang dikirimi |
| `alt` | **Alternative Fragment** | Blok kondisional — percabangan jika/maka/selain itu |
| `[kondisi]` | **Guard** | Kondisi penjaga yang harus terpenuhi agar blok dijalankan |

## A. Skenario 1: Registrasi, Verifikasi OTP, dan Inisialisasi Dompet

Alur ini memetakan proses keamanan awal sistem. Sistem mewajibkan verifikasi identitas melalui OTP via Gmail dan memastikan pengguna baru mendaftarkan dompet pertama sebelum dapat mengakses Dashboard.

```mermaid
sequenceDiagram
    actor User as User
    participant FE as Frontend React
    participant API as API Controller Laravel
    participant SVC as Mail Service
    participant DB as Database MySQL

    Note over User, DB: FASE 1 - Request OTP dan Registrasi Akun

    User->>FE: Mengisi form registrasi nama, email, password
    FE->>API: POST /api/register
    activate API
    API->>DB: INSERT ke tabel users status pending
    activate DB
    DB-->>API: User record berhasil dibuat
    deactivate DB
    API->>DB: INSERT kode OTP ke tabel otp_verifications
    activate DB
    DB-->>API: OTP tersimpan
    deactivate DB
    API->>SVC: Trigger kirim email OTP ke user
    activate SVC
    SVC-->>User: Email berisi kode OTP dikirim
    deactivate SVC
    API-->>FE: 200 OK - Instruksi masukkan OTP
    deactivate API
    FE-->>User: Tampilkan form input kode OTP

    Note over User, DB: FASE 2 - Verifikasi OTP

    User->>FE: Memasukkan kode OTP
    FE->>API: POST /api/verify-otp
    activate API
    API->>DB: SELECT kode OTP dari tabel otp_verifications
    activate DB
    DB-->>API: Kembalikan data OTP
    deactivate DB

    alt Kode OTP Tidak Valid atau Expired
        API-->>FE: 422 Unprocessable - Kode OTP salah
        FE-->>User: Tampilkan pesan error minta input ulang
    else Kode OTP Valid
        API->>DB: UPDATE status user menjadi active
        activate DB
        DB-->>API: Status diperbarui
        deactivate DB
        API->>DB: DELETE kode OTP dari tabel otp_verifications
        activate DB
        DB-->>API: OTP dihapus
        deactivate DB
        API->>SVC: Terbitkan Bearer Token via Laravel Sanctum
        activate SVC
        SVC-->>API: Token berhasil diterbitkan
        deactivate SVC
        API-->>FE: 200 OK - Bearer Token
        FE-->>User: Redirect ke halaman inisialisasi dompet
    end
    deactivate API

    Note over User, DB: FASE 3 - Inisialisasi Dompet Pertama

    User->>FE: Menginput nama dan saldo awal dompet
    FE->>API: POST /api/wallets dengan Bearer Token
    activate API
    API->>DB: INSERT ke tabel wallets balance sama dengan saldo awal
    activate DB
    DB-->>API: Dompet berhasil dibuat
    deactivate DB
    API-->>FE: 201 Created - Data dompet
    deactivate API
    FE-->>User: Redirect ke halaman Dashboard
```

**Penjelasan Alur:**

| # | Objek Pengirim | Objek Penerima | Pesan / Aksi |
|---|---|---|---|
| 1 | User | Frontend | Isi form registrasi (nama, email, password) |
| 2 | Frontend | API Controller | `POST /api/register` |
| 3 | API Controller | Database | `INSERT` data user (status: *pending*) |
| 4 | API Controller | Database | `INSERT` kode OTP ke `otp_verifications` |
| 5 | API Controller | Mail Service | Trigger kirim email OTP |
| 6 | Mail Service | User | Email OTP terkirim ke inbox |
| 7 | User | Frontend | Input kode OTP yang diterima |
| 8 | Frontend | API Controller | `POST /api/verify-otp` |
| 9 | API Controller | Database | Cocokkan kode OTP di database |
| 10 *(alt: gagal)* | API Controller | Frontend | `422` — OTP salah atau expired |
| 10 *(alt: berhasil)* | API Controller | Database | `UPDATE` status → `active`, `DELETE` OTP |
| 11 | API Controller | Mail Service (Sanctum) | Terbitkan Bearer Token |
| 12 | User | Frontend | Input nama dan saldo awal dompet |
| 13 | Frontend | API Controller | `POST /api/wallets` + Bearer Token |
| 14 | API Controller | Database | `INSERT` ke tabel `wallets` |
| 15 | Frontend | User | Redirect ke Dashboard |

## B. Skenario 2: Pencatatan Transaksi Pengeluaran *(Strict Balance Flow)*

Alur paling krusial yang melibatkan validasi bertingkat: pengecekan kecukupan saldo sebelum transaksi disimpan, dan evaluasi limit budget setelah transaksi berhasil. Seluruh operasi database bersifat atomik.

```mermaid
sequenceDiagram
    actor User as User
    participant FE as Frontend React
    participant API as API Controller Laravel
    participant DB as Database MySQL

    Note over User, DB: INPUT TRANSAKSI PENGELUARAN

    User->>FE: Mengisi form pengeluaran nominal, kategori, dan dompet
    FE->>API: POST /api/transactions dengan Bearer Token
    activate API

    Note over API, DB: VALIDASI INPUT DAN PENGECEKAN SALDO

    API->>DB: SELECT balance FROM wallets WHERE id sama dengan dompet dipilih
    activate DB
    DB-->>API: Kembalikan saldo terakhir dompet
    deactivate DB

    alt Saldo Kurang dari Nominal
        API-->>FE: 400 Bad Request - Saldo Tidak Cukup
        FE-->>User: Tampilkan pesan error saldo tidak cukup
    else Saldo Mencukupi
        Note over API, DB: DATABASE TRANSACTION ATOMIC
        API->>DB: BEGIN TRANSACTION
        API->>DB: INSERT ke tabel transactions nominal kategori wallet_id
        activate DB
        DB-->>API: Transaksi tersimpan
        deactivate DB
        API->>DB: UPDATE wallets SET balance = balance dikurangi nominal
        activate DB
        DB-->>API: Saldo dompet diperbarui
        deactivate DB
        API->>DB: SELECT SUM pengeluaran vs limit FROM budgets WHERE kategori
        activate DB
        DB-->>API: Data total pengeluaran dan limit budget kategori
        deactivate DB

        alt Pengeluaran Melebihi Limit Budget
            API->>DB: COMMIT TRANSACTION
            DB-->>API: Semua perubahan dipermanenkan
            API-->>FE: 201 Created - Data transaksi dengan flag is_overbudget true
            FE-->>User: Tampilkan notifikasi peringatan overbudget merah
        else Pengeluaran Masih Dalam Batas Budget
            API->>DB: COMMIT TRANSACTION
            DB-->>API: Semua perubahan dipermanenkan
            API-->>FE: 201 Created - Data transaksi sukses
            FE-->>User: Update grafik Dashboard secara real-time
        end
    end
    deactivate API
```

**Penjelasan Alur:**

| # | Objek Pengirim | Objek Penerima | Pesan / Aksi |
|---|---|---|---|
| 1 | User | Frontend | Isi form pengeluaran (nominal, kategori, dompet) |
| 2 | Frontend | API Controller | `POST /api/transactions` + Bearer Token |
| 3 | API Controller | Database | `SELECT balance` dari `wallets` |
| 4 *(alt: saldo kurang)* | API Controller | Frontend | `400 Bad Request` — saldo tidak cukup |
| 5 *(alt: saldo cukup)* | API Controller | Database | `BEGIN TRANSACTION` |
| 6 | API Controller | Database | `INSERT` record ke `transactions` |
| 7 | API Controller | Database | `UPDATE` kurangi saldo di `wallets` |
| 8 | API Controller | Database | `SELECT` total pengeluaran vs limit `budgets` |
| 9a *(overbudget)* | API Controller | Frontend | `201` + flag `is_overbudget: true` |
| 9b *(normal)* | API Controller | Frontend | `201` + data transaksi sukses |
| 10 | Frontend | User | Update grafik / tampilkan notifikasi |

## C. Skenario 3: Manajemen Profil & Pemrosesan Gambar *(WebP Processing Flow)*

Alur ini menjelaskan bagaimana sistem memproses foto profil yang diunggah pengguna: melakukan *crop* otomatis, konversi format ke WebP, menyimpan file ke storage, dan memperbarui database tanpa memerlukan *reload* halaman.

```mermaid
sequenceDiagram
    actor User as User
    participant FE as Frontend React
    participant API as API Controller Laravel
    participant IMG as Image Service Intervention Image
    participant STR as Local Storage
    participant DB as Database MySQL

    Note over User, DB: UPLOAD DAN PEMROSESAN FOTO PROFIL

    User->>FE: Memilih file gambar dari perangkat
    FE->>API: POST /api/profile/upload multipart/form-data dengan Bearer Token
    activate API

    Note over API, IMG: PEMROSESAN GAMBAR OTOMATIS

    API->>IMG: Kirim raw file ke Image Service
    activate IMG
    IMG->>IMG: Crop otomatis menggunakan center crop
    IMG->>IMG: Encode dan konversi ke format webp
    IMG-->>API: Kembalikan file WebP hasil pemrosesan
    deactivate IMG

    API->>STR: Simpan file ke storage/app/public/avatars/
    activate STR
    STR-->>API: File tersimpan, kembalikan path lengkap
    deactivate STR

    Note over API, DB: UPDATE DATABASE

    API->>DB: UPDATE users SET avatar_url = path_file_baru WHERE id = user_id
    activate DB
    DB-->>API: Kolom avatar_url berhasil diperbarui
    deactivate DB

    API-->>FE: 200 OK - Kembalikan URL avatar baru
    deactivate API
    FE-->>User: Ganti gambar profil tanpa reload halaman
```

**Penjelasan Alur:**

| # | Objek Pengirim | Objek Penerima | Pesan / Aksi |
|---|---|---|---|
| 1 | User | Frontend | Pilih file gambar dari perangkat |
| 2 | Frontend | API Controller | `POST /api/profile/upload` multipart/form-data |
| 3 | API Controller | Image Service | Kirim *raw file* untuk diproses |
| 4 | Image Service | Image Service | *Self-message*: crop otomatis (center) |
| 5 | Image Service | Image Service | *Self-message*: encode ke `.webp` |
| 6 | Image Service | API Controller | Kembalikan file WebP hasil proses |
| 7 | API Controller | Local Storage | Simpan ke `storage/app/public/avatars/` |
| 8 | Local Storage | API Controller | Kembalikan path file tersimpan |
| 9 | API Controller | Database | `UPDATE avatar_url` di tabel `users` |
| 10 | API Controller | Frontend | `200 OK` + URL avatar baru |
| 11 | Frontend | User | Tampilkan foto profil baru tanpa reload |

## D. Skenario 4: Transfer Antar Dompet *(Atomic Transaction Flow)*

Alur ini menjelaskan pemindahan saldo antar dompet menggunakan mekanisme *database transaction* atomik. Jika salah satu operasi gagal di tengah proses, seluruh transaksi dibatalkan (*rollback*) untuk menjaga konsistensi saldo.

```mermaid
sequenceDiagram
    actor User as User
    participant FE as Frontend React
    participant API as API Controller Laravel
    participant DB as Database MySQL

    Note over User, DB: INPUT DATA TRANSFER

    User->>FE: Memilih Dompet Asal, Dompet Tujuan, dan Nominal transfer
    FE->>API: POST /api/transfer dengan Bearer Token
    activate API

    Note over API, DB: MULAI DATABASE TRANSACTION ATOMIK

    API->>DB: BEGIN TRANSACTION via DB beginTransaction
    API->>DB: SELECT balance FROM wallets WHERE id = dompet_asal FOR UPDATE
    activate DB
    DB-->>API: Kembalikan saldo Dompet Asal
    deactivate DB

    alt Saldo Dompet Asal Tidak Mencukupi
        API->>DB: ROLLBACK semua perubahan
        API-->>FE: 400 Bad Request - Saldo Dompet Asal Tidak Cukup
        FE-->>User: Tampilkan pesan error transfer gagal
    else Saldo Dompet Asal Mencukupi
        API->>DB: UPDATE wallets SET balance = balance - nominal WHERE id = dompet_asal
        activate DB
        DB-->>API: Saldo Dompet Asal berhasil dikurangi
        deactivate DB

        API->>DB: UPDATE wallets SET balance = balance + nominal WHERE id = dompet_tujuan
        activate DB
        DB-->>API: Saldo Dompet Tujuan berhasil ditambahkan
        deactivate DB

        API->>DB: INSERT ke tabel transfers from_wallet to_wallet nominal timestamp
        activate DB
        DB-->>API: Log transfer tersimpan
        deactivate DB

        API->>DB: COMMIT TRANSACTION semua perubahan dipermanenkan
        activate DB
        DB-->>API: Commit berhasil
        deactivate DB

        API-->>FE: 201 Created - Data transfer berhasil
        FE-->>User: Update saldo kedua dompet di Dashboard secara real-time
    end
    deactivate API
```

**Penjelasan Alur:**

| # | Objek Pengirim | Objek Penerima | Pesan / Aksi |
|---|---|---|---|
| 1 | User | Frontend | Pilih dompet asal, tujuan, dan nominal |
| 2 | Frontend | API Controller | `POST /api/transfer` + Bearer Token |
| 3 | API Controller | Database | `BEGIN TRANSACTION` |
| 4 | API Controller | Database | `SELECT balance` Dompet Asal (`FOR UPDATE`) |
| 5 *(alt: gagal)* | API Controller | Database | `ROLLBACK` — batalkan semua |
| 5 *(alt: berhasil)* | API Controller | Database | `UPDATE` kurangi saldo Dompet Asal |
| 6 | API Controller | Database | `UPDATE` tambah saldo Dompet Tujuan |
| 7 | API Controller | Database | `INSERT` log ke tabel `transfers` |
| 8 | API Controller | Database | `COMMIT TRANSACTION` |
| 9 | API Controller | Frontend | `201 Created` — transfer berhasil |
| 10 | Frontend | User | Tampilkan saldo terbaru kedua dompet |
