### 3.4 Batas Perancangan (Design Constraints)

Batas perancangan menentukan batasan teknis dan metodologi yang harus dipatuhi dalam pengembangan perangkat lunak **Sapopoe**. Batasan ini ditetapkan untuk menjamin standarisasi kode, keamanan data finansial, serta kemudahan dalam pemeliharaan sistem jangka panjang oleh Tim REMAKO.

#### 3.4.1 Standar Bahasa dan Framework
* **Backend Logic:** Pengembangan sisi server wajib menggunakan bahasa pemrograman **PHP 8.2+** dengan framework **Laravel 10/11**. Penggunaan fungsi PHP murni (*native*) tanpa melalui standar framework sangat tidak disarankan untuk menjaga konsistensi struktur.
* **Frontend Logic:** Antarmuka pengguna harus dibangun menggunakan library **React.js** dengan sistem state management yang efisien (seperti *React Hooks* atau *Zustand*).
* **Coding Standard:** Penulisan kode harus mengikuti standar **PSR-12** untuk PHP dan standar **ESLint** untuk JavaScript guna memastikan keterbacaan kode antar anggota tim.

#### 3.4.2 Arsitektur dan Data
* **RESTful Architecture:** Komunikasi antara klien dan server harus melalui *End-point* API yang stateless. Setiap permintaan wajib menyertakan *Authorization Header* dengan *Bearer Token* setelah pengguna terautentikasi.
* **Integritas Database:** Perancangan basis data wajib menggunakan mesin **InnoDB** pada MySQL untuk mendukung fitur *Foreign Key* dan *Transactions*. Hal ini krusial untuk memastikan bahwa jika pencatatan transaksi gagal, saldo dompet tidak akan berubah (atomisitas).
* **Single Role Policy:** Rancangan sistem tidak mendukung fitur *multi-tenancy* atau manajemen admin. Seluruh logika desain difokuskan pada isolasi data per individu pengguna.

#### 3.4.3 Keamanan dan Enkripsi
* **Hashing:** Semua data sensitif, terutama kata sandi, wajib menggunakan algoritma **Bcrypt** dengan *work factor* minimal 12.
* **CORS Policy:** Server hanya diperbolehkan menerima permintaan dari alamat domain/IP yang telah didaftarkan dalam *whitelist* konfigurasi CORS untuk mencegah serangan *Cross-Site Request Forgery* (CSRF).
* **Validation Layer:** Validasi data wajib dilakukan di dua sisi ( *Double Validation*). Validasi di sisi klien (React) untuk pengalaman pengguna, dan validasi di sisi server (Laravel Request Validation) sebagai pertahanan utama.

#### 3.4.4 Manajemen Aset Digital
* **Image Processing:** Sistem tidak diperbolehkan menyimpan file gambar asli yang diunggah pengguna. Setiap gambar wajib diproses melalui library *Intervention Image* untuk di-*resize* dan dikompresi ke format **WebP** guna menjaga efisiensi ruang penyimpanan dan kecepatan *loading* halaman.
* **Storage Path:** Aset publik seperti foto profil tidak boleh diakses secara langsung melalui jalur folder internal, melainkan melalui *symbolic link* (`storage:link`) untuk menjaga keamanan struktur folder inti aplikasi.

#### 3.4.5 Kepatuhan Waktu dan Jadwal
* **Inactivity Rule:** Rancangan sistem harus menyertakan mekanisme otomatis yang mengeksekusi penghapusan data secara permanen jika akun terdeteksi tidak aktif selama 30 hari kalender, sesuai dengan kebijakan manajemen data mandiri Sapopoe.
