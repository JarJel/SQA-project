### 2.4 Batasan-Batasan (Constraints)

Pengembangan dan operasionalisasi perangkat lunak **Sapopoe** dibatasi oleh sejumlah parameter teknis, regulasi, dan arsitektur untuk memastikan keamanan, keandalan, dan fokus proyek. Batasan-batasan ini wajib dipatuhi oleh seluruh anggota Tim REMAKO selama fase pengembangan dan SQA (*Software Quality Assurance*).

Berikut adalah batasan-batasan sistem yang mendasari aplikasi Sapopoe:

#### 1. Batasan Teknologi dan Framework (Technological Constraints)
* **Frontend:** Pengembangan antarmuka pengguna diwajibkan menggunakan **React.js** dengan pendekatan SPA (*Single Page Application*) untuk memberikan pengalaman navigasi yang mulus tanpa *reload*.
* **Backend:** Logika bisnis dan penyediaan API wajib dibangun menggunakan *framework* **Laravel (PHP)**.
* **Database:** Sistem harus menggunakan **MySQL** sebagai RDBMS utama, dengan pemodelan relasi antar-tabel (*Foreign Keys*) yang ketat untuk menjaga integritas data keuangan (misal: *Cascade On Delete* pada relasi *User* dan *Transaction*).

#### 2. Batasan Keamanan Data (Security Constraints)
* **Penyimpanan Kata Sandi:** Dilarang keras menyimpan kata sandi pengguna dalam bentuk *plain text*. Semua *password* wajib dienkripsi menggunakan algoritma *Bcrypt* bawaan Laravel.
* **Sesi Akses (Session & Token):** Autentikasi API harus menggunakan **Laravel Sanctum**. Token akses akan kadaluarsa jika pengguna melakukan *Logout* atau akun terkena penghapusan otomatis (*Inactivity Deletion*).
* **Validasi Sisi Server (Backend Validation):** Semua input keuangan dari klien (React) dilarang langsung dimasukkan ke basis data. Wajib melewati lapisan validasi *Controller* di Laravel untuk mencegah injeksi SQL dan input negatif pada saldo/bunga.

#### 3. Batasan Operasional dan Integrasi (Operational Constraints)
* **Isolasi Finansial:** Aplikasi ini tidak memiliki kewenangan dan kemampuan teknis untuk memindahkan uang sungguhan. Sapopoe murni berfungsi sebagai alat pencatatan (*Tracker*). Tidak ada integrasi dengan gerbang pembayaran (*Payment Gateway*) seperti Midtrans atau Xendit.
* **Standarisasi Media:** Foto profil (Avatar) pengguna yang diunggah dibatasi maksimal berukuran **5MB**. Sistem secara *hard-code* akan memotong (*crop*) dan mengkonversi gambar tersebut menjadi format **WebP** resolusi 300x300 pixel untuk meminimalisir beban memori server.

#### 4. Batasan Kompatibilitas Antarmuka (Interface Constraints)
* Desain aplikasi difokuskan pada pengujian di peramban (*browser*) berbasis Chromium (Google Chrome, Microsoft Edge) dan WebKit (Safari). Kompatibilitas mundur (*backward compatibility*) untuk peramban lawas seperti Internet Explorer 11 **tidak didukung**.
